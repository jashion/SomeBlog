#iOS观察者模式学习

###（1）KVC与KVO简介

  • KVC

KVC（Key-value coding）是一种间接更改对象状态的方式。

官方文档描述：

Key-value coding is a mechanism for accessing an object’s properties indirectly, using strings to identify properties, rather than through invocation of an accessor method or accessing them directly through instance variables. In essence, key-value coding defines the patterns and method signatures that your application’s accessor methods implement.

  • KVC的作用

通过键值的方式访问对象或者修改对象的值

比如：NSString *name = [car valueForKey: @”name”];通过-valueForKey:获取对象，它会先查找以参数名命名（格式为-key或-isKey）的getter方法，对于上面的调用，valueForKey:会先寻找-name方法。如果没有这样的getter方法，它将会在对象内寻找名称格式为_key或key的实例变量。通过使用KVC，没有相关getter方法也能获取对象，不需要通过对象指针来直接访问实例变量。

  • KVC的优缺点

1.KVC可以轻松处理集合类。（如：NSArray）

2.KVC没有相关getter方法也能获取对象，不需要通过对象指针来直接访问实例变量

3.简化代码

4.KVC需要解析字符串，速度比较慢

5.编译器无法进行错误检查

  • KVO

KVO（Key-value Observing）是基于KVC的，一种监听方式，也就是所谓的观察者模式。

官方文档描述：

Key-value observing provides a mechanism that allows objects to be notified of changes to specific properties of other objects. It is particularly useful for communication between model and controller layers in an application.

#如何理解iOS的多线程

###前言：

前段时间，很想静下心来研究一下YYText，看了ibireme的一篇《iOS保持界面流畅的技巧》的博客，下了源码，发现里面用到多线程，异步渲染，异步记载，悲催的是，自己对于多线程并不很熟悉，只是在公司项目里面用过轻量级的GCD，所以，下定决心来补一下iOS多线程的知识。

###1.先弄清楚和线程有关的几个概念（大学读书不用功的同学伤不起T_T）

进程：一个包含多个线程的可执行文件。（The term process is used to refer to a running executable, which can encompass multiple threads.）

线程：一段可以执行的对立代码。（The term thread is used to refer to a separate path of execution for code.）

任务：对于所做工作的抽象概念。（The term task is used to refer to the abstract concept of work that needs to be performed.）

队列：我的理解是，一个装载任务的容器。

同步：在当前的线程执行任务，会阻塞当前线程的操作。

异步：另开一个新线程执行任务，不会阻塞当前的线程操作。

串行：队列的任务，根据FIFO的规则，一个个执行。

并行：队列的任务，并行执行。如果CPU是多核，则会多个线程一齐执行，如果非多核，则会采取时间片轮转的调度方式，根据FIFO的规则执行，由于没个线程执行的时间很短，可以看成是同步执行的队列。

####进程，线程，任务和队列的区别和联系：

（1）进程可以看作一个程序，可以理解成一个App就是一个进程，一个进程包括一个或者多个线程。

（2）线程则是一个进程里面的概念，同一个进程包含的线程可以共享进程里面的数据，所以，多线程安全的问题也就突显出来了。线程里面的数据不共享，但线程和线程之间可以通信。

（3）任务只是一个抽象的概念，一个线程里面的某个有意义的代码片段可以称之为任务，多个线程完成一个工作也可以称之为任务，所以，任务和线程不是同一层面的意思。

（4）队列，可以简单的理解为，一个容纳任务的容器。

###2.iOS里面可用的多线程

  • PThread(现实中不怎么用到，所以略过)
  • NSThread
  • GCD(Grand Central Dispatch)
  • NSOperation

下面我会一一讲解，我所了解到的多线程。

NSThread基于Objective-C对象，是苹果用来取代PThread的，方便使用，但是需要手动管理线程的生命周期(线程创建，调度和销毁等)。NSThread相比于GCD，NSOperation更趋向于系统级别的多线程的封装，使用起来没有GCDN和SOperation那么方便，但是，也有一些好用的功能，比如获取当前线程的相关信息：[NSThread currentThread];所以，多线程开发使用哪种实现，取决于你需要实现功能的情景，一般比较轻量级的，好用的就选GCD.GCD是苹果开发的，基于C语言的多线程，不需要手动管理线程的生命周期，而且它可以充分利用多核的优势。如果只想新开一个线程运行一些函数而不关心其他的东西，你可以选择GCD；但是如果你想实现更复杂的行为，那么应该选择NSOperation，因为NSOperation是基于GCD（以前基于NSThread）的更高层次的API，并且功能比CGD更强大，自由度更高，可以实现更复杂的线程依赖。

#浅谈iOS内存管理

###1.MRC(Manul Reference Counting)人工引用计数

在ARC出现之前，需要手动管理内存。如何手动管理内存呢？首先，得知道，在iOS系统里面，对象的生命周期是通过引用计数管理的。每个对象 都有着一个与之关联的整数。当使用alloc,new方法或者通过copy消息创建对象的时候，对象的保留计数器的值设置为1。当对象调用retain方法，该对象的保留计数增加1，反之，如果对象调用release方法，该对象的保留计数减少1（注意：如果对象本身的保留计数为0，再调用或者说发送release消息就会造成crash）。当一个对象的计数为0时，系统会自动向对象发送dealloc消息，销毁对象，释放内存。

手动管理内存需要注意下面几点：

（1）使用alloc,new或copy方法创建对象时，该对象的保留计数为1。当不再使用该对象时，向该对象发送一条release或者autorelease消息。

（2）如果你通过其他方法获得一个对象，则不需要你来管理该对象的生命周期，除非你想在一段时间内拥有改对象，则需要在使用前retain，使用完成后release。

（3）必须保证retain和release成对出现。

（4）release操作只是使当前对象的保留计数减1，并不是销毁该对象。当该对象的保留计数为0时，系统才会发送dealloc消息，销毁对象。

（5）把对象加入autorelease也不是立马使该对象的保留计数减1，而是在未来适当的时机release该对象。

###2.ARC（Automatic reference counting）自动引用计数

不得不说，苹果引入ARC是一个英明的决定，从此，苹果开发者告别手动管理内存的艰难时期，是广大开发者的福音。为什么ARC那么强大呢？其实ARC只是做了一件事，就是在编译时自动帮你的代码插入retain和release。恰恰，它强大之处就在于这里，避免开发者忘记添加retian和release，一两个对象不觉得的，如果对象很多的话，就很容易忘记，所以说，ARC帮我们解决了手动管理对象生命周期的痛苦，让我们更专注于代码的逻辑实现。

虽然ARC很强大，但是并不是所有的对象都能用ARC,一些不能使用ARC的对象，还是得手动管理内存。

可以使用ARC的对象：

（1）block

（2）Objective-C对象

（3）_attribute((NSObject))修饰的对象

说说strong,weak,retain,assign,copy修饰符：

（1）strong和weak

以前一直不明白，strong和weak修饰变量，有什么用？网上的说法大多都是雷同，就是强引用和弱引用关系。现在才明白，其实强引用指的是修饰的变量指向的对象的关系，也就相对于对象的强引用和弱引用。比如，对象A用strong修饰，对象B用weak修饰，那么A引用B就是强引用，B引用A就是弱引用。strong和weak的区别在于对对象的束缚力。stackOver上有个比较有趣的例子形容strong和weak，就是在某棵树下有一条狗，strong就是那些拿着拴狗链的人，只要还有人在，狗就跑不了，而weak是没有拴狗链的人，只是语言上的规劝，要狗好好的呆着别动，但是狗还是可以跑的。总之一句话，有强哥在准跑不了。

（2）strong和retain

strong和retain差不多，没啥区别，只不过ARC以后都习惯用strong。

（3）weak和assign

weak和assign最大的区别在于，weak修饰的对象保留计数为0，则把该对象设置为nil，assign不会。设置为nil的好处在于，如果误发消息给该对象，不会crash，因为nil也是属于对象，如果是null就会crash。

###3.ARC与autorelease
ARC的作用就是在编译时期帮代码自动插入retain和release，而autorelease就是将需要释放的对象加入到一个stack里面，并不会立刻释放，而是在autorelease pool销毁时，一齐释放栈里面的对象。其实就是相当于release的作用。当时，不明白，既然有了ARC为什么还需要autoreleasepool？其实关键之处在于作用域，如果的创建与释放隔的很近，ARC在编译时就很容易判断在哪个位置插入retain和release，但是如果横跨几个文件或者说不知道什么时候释放该对象，这时候autorelease就很有用了，每个线程的RunLoop都有自己的autoreleasepool，当该RunLoop结束时，autoreleasepool就会被销毁，里面所有的对象也会一一销毁。
